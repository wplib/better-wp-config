> "I do have some minor problems with the locations of the files which I assume are easily solvable"

That is fully configurable in wp-bootstrap.php.  Just specify a different directory for the config.  The structure inside that directory is currently fixed, but Better WP-Config (BWPC) is a proof-of-concept at the moment and open for feedback and potential improvement from others.  I had to decide a bunch of things which were arbitrary and we could easily select different approaches for those arbitrary choices.
3:35 PM
> "IMHO for "project it should be the .env format"

First argument for .php being the default is that BWPC can work with .php format without anything else being installed.  It requires phpdotenv to be installed with Composer to use .env, so that right there is one reason for .php being the default.  Ultimately we could roll-our-own .env support and eliminate the dependency but 1.) I think people wanting to use .env will likely be more advanced and thus be Composer users, and 2.) read on as to why I am skeptical about .env actually being a better solution than .php.
Ironically the .env format has limitations compared with the currently default .php format.  Take are look at this Pantheon configuration file and consider how you would do this in .env?
- https://github.com/wplib/better-wp-config/blob/master/wp-content/config/providers/pantheon.php
That Pantheon "provider" in the link above basically duplicates what they Pantheon has as their defaults:
- https://github.com/pantheon-systems/WordPress/blob/default/wp-config.php
So as I work with BWPC and web hosts like Pantheon that already use $_ENV for DB credentials the .env format seems to be redundant and thus solving a non-problem except on local development (and maybe shared hosting?), but VLucas says (emphasis mine): (edited)
3:35 PM
> "phpdotenv is made for development environments, and generally should not be used in production.

- From https://github.com/vlucas/phpdotenv#usage-notes
And on local development you would use non-secrets for DB so no problem in committing those to version control.
And even if you did need to use secrets like API keys the problem is the same if you are using .env or .php.
Maybe we need to add a config-secrets.php loaded after the environment but before the host which you always .gitignore and never commit to version control?
So I'm questioning if the growing consensus that we should all move to .env files for WordPress-ish (X-Press?) implementations considers all the ramifications of said use?  At this point I am thinking if people are given the choice of using .env and .php they will fall back to the .php because AFAICT it makes necessary things possible that you can't (easily?) do with .env. So  I am coming to believe it has been a collective "Seemed like a good idea at the time" that people will move on from when given the actual option of using it for X-Press sites.
But I am more than willing to be proven wrong on this, if I am in fact missing something.
I think that what needs to happen is numerous people need to try to use it, both as .env and as .php before we can make a final decision on which is the preferred default. (edited)
